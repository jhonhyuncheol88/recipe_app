## 1. 리포트 페이지 후속 (재고 가치 제거 + 색상 + 월 선택)

### 동기
- 첫 부팅에서 `RecipePriceHistoryRepository` 미등록으로 `ReportCubit` 생성
  실패 → "Bad state: provider that threw during the creation" 트리거.

  `model/unit.dart::Unit` ↔ `util/unit_converter.dart::Unit` 캐스트가 실패

  throw → UI 멈춤.

## 2. 앱 오픈 광고 도입

### 동기
- cold-start 시 사용하던 전면(interstitial) 광고를 App Open 광고로 교체.
- resume(다른 앱에서 복귀) 시에는 노출하지 않는다 (cold-start 1회).
- `.env` 에 4개 키 추가됨: `ADMOB_ANDROID_APP_OPEN_ID`,
  `ADMOB_IOS_APP_OPEN_ID`, `ADMOB_ANDROID_APP_OPEN_ID_TEST`,
  `ADMOB_IOS_APP_OPEN_ID_TEST`.

### 변경
- **NEW** `lib/service/app_open_ad_service.dart` — `AppOpenAdService` 싱글톤.
  - `getAppOpenAdUnitId()` — **디버그 빌드는 무조건 `*_TEST` env 키만 사용**,
    없으면 Google sample test ID 로 fallback. 릴리즈 빌드만 prod 키 호출.
    `ADMOB_FORCE_PRODUCTION` 분기는 두지 않음 (운영 광고 오염 방지).
  - `loadAppOpenAd()` — `Completer<AppOpenAd?>? _loadCompleter` 로 진행 중인
    로드를 공유. `preload()` 와 `showIfAvailable()` 가 동시에 호출돼도 같은
    future 를 await → "이미 로드 중 — skip" 으로 인한 광고 누락이 사라짐.
  - `preload()` — 캐시 비어있고 표시 중도 아닐 때만 `_preloadedAd` 채움.
  - `showIfAvailable()` — Premium gate (`PremiumCubit.state.isPremium`) 체크
    후 표시. 닫힘 시 `unawaited(preload())` 로 다음 광고 미리 로드.
- **MODIFY** `lib/main.dart`
  - `import 'service/app_open_ad_service.dart';`
  - `_postAppInitialization` — 이전 `if (Platform.isAndroid)` iOS 가드 제거
    → iOS/Android 양쪽 모두 `AdMobForwardService.initialize()` + App Open
    preload + 표시.
  - `_showAppOpenAdWithCooldown` 안 `showInterstitialAd()` 호출을
    `AppOpenAdService.showIfAvailable()` 로 교체. 10분 쿨다운 prefs 키
    (`last_ad_shown_time_millis`) 는 그대로 유지.
  - PremiumCubit 생성 시 `AppOpenAdService.setPremiumGate(...)` 도 함께 등록.
- **MODIFY** `lib/service/admob_forward.dart`
  - `initialize()` 안의 자동 `_preloadInterstitialAdInBackground()` 호출 제거.
    전면 광고는 이제 부팅 시 받지 않고, `showInterstitialAd()` 호출 시점에
    lazy 로드 + 표시 직후에만 다음 광고를 백그라운드 preload.

### 알려진 환경 의존
- iOS `Info.plist` 의 `GADApplicationIdentifier` (line 38) +
  `SKAdNetworkItems` (line 60) 는 Google sample 값으로 셋업되어 있음 (실제
  배포 전 Apple 콘솔 ID 로 교체 필요).
- 운영 광고 노출 전 `NSUserTrackingUsageDescription` (ATT) 추가 권장.
- 시뮬레이터에서 광고 SDK 가 종종 응답 지연/no-fill → cold-start 노출 실패는
  정상. 실기기 검증 권장.

## 3. 부팅 시퀀스 직렬화

### 동기
- 기존 main 의 `addPostFrameCallback` 과 `PermissionRequester` 의 별도
  `addPostFrameCallback` 이 동시에 동작하면서 알림 권한 다이얼로그 + App Open
  광고 + 인앱 리뷰가 거의 같은 순간에 native present → iOS 가 큐잉 또는
  hang detector 트리거 (`bug_type: 309`, main thread hang report).
- 사용자 요구 순서: **알림 권한 → 온보딩 → 앱 열기 광고 → (리뷰는 별도)**.

### 변경
- **MODIFY** `lib/main.dart`
  - 전역 `Completer<void> _notificationReadyCompleter` 추가 — 알림 권한 단계
    의 게이트.
  - 기존 직렬 (`_postAppInitialization → _initializeInitialData →
    _checkAndRequestReview`) 제거하고 단일 시퀀서 `_runStartupSequence` 로
    교체:
    1. `await _notificationReadyCompleter.future` (알림 권한)
    2. `await _waitUntilOnboardingCompleted()` (`onboarding_completed` prefs
       polling, 1초 간격, 이미 완료 사용자는 즉시 통과)
    3. `await _initializeInitialData(logger)`
    4. `await _postAppInitialization(logger)` (AdMob init + App Open 광고)
  - `_checkAndRequestReview` 함수 + `in_app_review_service.dart` import 삭제.
- **MODIFY** `lib/main.dart` — `PermissionRequester._initializeNotificationService`
  의 `finally` 에서 `_notificationReadyCompleter.complete()` 호출. 성공/실패
  어느 경로에서도 게이트가 풀려 영원히 대기하지 않음.

### 효과
- 권한 다이얼로그가 떠 있는 동안 광고가 동시에 present 되는 native 충돌
  제거.
- 온보딩 진행 중 광고/리뷰가 끼어들지 않음.
- 전면 광고 cold-start preload 가 빠지면서 main isolate 부하 감소 → hang
  detector 트리거 가능성 감소.

## 4. 인앱 리뷰 트리거 시점 변경

### 동기
- 부팅 직후 무조건 리뷰 다이얼로그 호출 → 사용자가 앱의 가치를 인지하기
  전 노출되어 적합도 낮음.
- 정책: **재료 또는 레시피가 실제 생성된 시점**에 OS 다이얼로그 호출. OS
  rate limit (iOS: 4개월/3회, Android: 자체 throttling) 이 빈도 자동 조절.

### 변경
- **MODIFY** `lib/controller/ingredient/ingredient_cubit.dart`
  - `dart:async`, `service/in_app_review_service.dart` import.
  - `addIngredient` Analytics 호출 직후
    `unawaited(InAppReviewService().requestReview())`.
- **MODIFY** `lib/controller/recipe/recipe_cubit.dart`
  - 동일 패턴, `addRecipe` Analytics 호출 직후 한 줄.
- **MODIFY** `lib/main.dart` — 부팅 시퀀스에서 리뷰 단계 제거 (위 §3 참조).

### 동작
- `InAppReviewService.requestReview()` 의 `canRequestReview()` 가 명시 거부/
  완료 prefs (`user_declined`, `user_reviewed`) 만 스킵. 그 외엔 OS 다이얼로그
  호출. 표시 빈도 제어는 OS 에 위임.
